## 1. 全体像

### A: tmux-agents-starter（本リポジトリ）

**2つのオーケストレーションシステムを内蔵した実験場。**

```
┌─────────────────────────────────────────────────────────────┐
│                    tmux-agents-starter                       │
│                                                             │
│  ┌──────────────────────┐  ┌─────────────────────────────┐  │
│  │  System A: scripts/  │  │  System B: .yamibaito/      │  │
│  │  ──────────────────  │  │  ────────────────────────   │  │
│  │  ロールベース        │  │  ボス・クルーモデル        │  │
│  │  パイプライン型      │  │  ヤクザ階層型              │  │
│  │  orchestrator.js     │  │  YAML キュー + send-keys   │  │
│  │  8ロール逐次実行     │  │  親分→若頭→若衆 並列実行  │  │
│  └──────────────────────┘  └─────────────────────────────┘  │
│                                                             │
│  + git worktree による複数チーム同時稼働                     │

## 3. 設計思想の根本的な違い

| 観点 | A: tmux-agents-starter | B: dark-part-time-job |
|------|------------------------|------------------------|
| **設計哲学** | 実験場。複数アプローチを並行探索 | 汎用ツール。1つの方法をどこでも使う |
| **配置モデル** | リポジトリに組み込み（embedded） | 外部から注入（inject） |
| **オーケストレーション数** | 2つ（パイプライン＋階層型） | 1つ（階層型のみ） |
| **ソフトウェア開発プロセス** | 要件→設計→品質→実装→レビューのフルライフサイクル | タスク実行に特化。計画は別フェーズ |
| **品質保証** | 品質アーキテクト + 2段階レビュー | ペルソナベースの品質管理 |
| **LLMの多様性** | Claude / Codex / Copilot 混在プリセット | Claude (親分/若頭) + Codex (若衆) |
| **スケーラビリティ** | git worktree で複数チーム | session ID で複数セッション |
| **再利用性** | このリポジトリ専用 | どのリポジトリにも注入可能 |

| **設計プロセス** | 1st/2nd アーキテクト | SPEC.md テンプレート | **A** |
| **品質ゲート** | quality-architect | なし（ペルソナで代替） | **A** |
| **コードレビュー** | 2段階レビュー + 差し戻し | なし | **A** |
| **並列タスク実行** | 若衆5名並列 | 若衆3〜5名並列 | 引き分け |
| **汎用性（他リポジトリ対応）** | 困難（組み込み型） | `yb init` で即注入 | **B** |
| **CLI の充実度** | tmux-agents.sh のみ | yb CLI（init/start/stop/plan/collect等） | **B** |
| **計画バリデーション** | なし | `yb plan-review`（静的+LLM） | **B** |

| **Living Spec** | 2nd-reviewer が維持 | なし | **A** |
| **ダッシュボード** | dashboard.md (若頭更新) | dashboard.md (yb collect) | 引き分け |
| **レース条件防止** | RACE-001 ルール | 記載なし | **A** |
| **ペルソナシステム** | ロール固定 | 14種のペルソナ動的割当 | **B** |
| **セットアップの手軽さ** | clone → 設定必要 | `yb init` 一発 | **B** |

## 5. 階層構造の比較

### A: tmux-agents-starter

```
組長（ユーザー）
 │
 ├── System A: パイプライン型（逐次、品質重視）
 │    │
 │    planner ─→ 1st-arch ─→ 2nd-arch ─→ quality-arch

 │                              executor (Rally) ←┘
 │                                 │
 │                              2nd-reviewer
 │                                 │

      │

      若頭 (Claude Opus/Sonnet)
       │
      ├── 銀次 (worker_001)
      ├── 龍   (worker_002)

### B: dark-part-time-job

```
組長（ユーザー）

## 6. 通信プロトコルの違い

### A: System A — JSON シグナル方式

```
[エージェント] ──出力──→ [tmux pane stdout]
                              │
                    orchestrator.js が capture-pane
                              │
                    JSON ブロック検出 {"mission":"completed",...}
                              │
                    次のロールペインへ send-keys
```

- **メリット**: エージェントは JSON を吐くだけ。通信を意識しない
- **デメリット**: orchestrator.js が5秒ポーリング。レイテンシあり

### A: System B + B: dark-part-time-job — YAML キュー方式

```
[親分] ──Write──→ queue/director_to_planner.yaml
       ──send-keys──→ [若頭ペイン]

[若頭] ──Read──→ queue/director_to_planner.yaml
       ──Write──→ queue/tasks/worker_NNN.yaml
       ──send-keys──→ [若衆ペイン]

[若衆] ──Read──→ queue/tasks/worker_NNN.yaml
       ──Write──→ queue/reports/worker_NNN_report.yaml
       ──send-keys──→ [若頭ペイン]（yb run-worker が自動送信）

[若頭] ──yb collect──→ dashboard.md
       ──send-keys──→ [親分ペイン]
```

- **メリット**: ファイルベースで永続化。デバッグしやすい
- **デメリット**: ファイル I/O のオーバーヘッド

### 通信方式比較

| 比較軸 | JSON シグナル (A-System A) | YAML キュー (A-System B / B) |
|--------|---------------------------|------------------------------|
| 永続性 | 揮発（stdout のみ） | 永続（ファイルに残る） |
| デバッグ | capture-pane ログ頼り | YAML ファイルを直接確認可能 |
| レイテンシ | 5秒ポーリング | send-keys 即時 |
| 信頼性 | JSON パースエラーのリスク | YAML スキーマで構造保証 |
| 拡張性 | orchestrator.js の改修必要 | YAML にフィールド追加で対応 |

## 7. ユースケース別おすすめ

### この場合は A: tmux-agents-starter

| ユースケース | 理由 |
|-------------|------|
| **本格的なソフトウェア開発** | 要件→設計→品質→実装→レビューのフルパイプライン |
| **品質が最優先のプロジェクト** | 品質アーキテクト + 2段階レビュー + 差し戻しループ |
| **複数LLMを使い分けたい** | Claude/Codex/Copilot の混在プリセット |
| **長期プロジェクトの継続開発** | フィードバックループ + Living Spec |
| **複数チーム同時並行開発** | git worktree で完全分離 |
| **設計ドキュメントを重視** | アーキテクトロールが設計書を作成・維持 |

### この場合は B: dark-part-time-job

| ユースケース | 理由 |
|-------------|------|
| **既存リポジトリに素早く導入** | `yb init --repo /path` 一発 |
| **複数リポジトリを横断管理** | 中央オーケストレータから注入 |
| **タスク実行メイン（設計済み前提）** | 計画→実行に特化 |
| **軽量な並列作業** | 最小構成で並列ワーカー起動 |
| **計画フェーズの形式化** | PRD/SPEC/tasks.yaml + バリデーション |
| **新規プロジェクトの立ち上げ** | テンプレートから即構築 |
| **ペルソナの柔軟な割り当て** | 14種のペルソナを動的選択 |

### 両方使うべきケース

| ユースケース | 組み合わせ方 |
|-------------|------------|
| **大規模プロジェクト** | B で計画→A の System A で品質重視の実装 |
| **段階的な導入** | B で小さく始めて→A に移行して本格化 |
| **タスクの性質が混在** | 設計が必要なものは A、バッチ作業は B |

## 8. 強み・弱み分析

### A: tmux-agents-starter

```
  ┌─────────────────────────────────────────────┐
  │              強み (Strengths)                 │
  │                                              │
  │  ✦ フルライフサイクル対応                     │
  │  ✦ 品質保証が厚い（4チェックポイント）        │
  │  ✦ マルチLLM対応（5プリセット）               │
  │  ✦ フィードバックによる継続学習               │
  │  ✦ Living Spec の自動維持                     │
  │  ✦ git worktree で並行開発                    │
  │  ✦ 2つのシステムを場面で使い分け             │
  └─────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────┐
  │              弱み (Weaknesses)                │
  │                                              │
  │  ▽ セットアップが複雑                         │
  │  ▽ このリポジトリに密結合                     │
  │  ▽ 他リポジトリへの移植が困難                 │
  │  ▽ 2つのシステム並存で認知コスト高い          │
  │  ▽ orchestrator.js の5秒ポーリング            │
  │  ▽ CLI ツール (yb) が外部依存                 │
  │  ▽ 計画バリデーション機能がない               │
  └─────────────────────────────────────────────┘
```

### B: dark-part-time-job

```
  ┌─────────────────────────────────────────────┐
  │              強み (Strengths)                 │
  │                                              │
  │  ✦ どのリポジトリにも注入可能                 │
  │  ✦ yb CLI が充実（init/start/stop/plan等）   │
  │  ✦ 計画バリデーション内蔵                     │
  │  ✦ セットアップが簡単                         │
  │  ✦ 単一システムでシンプル                     │
  │  ✦ ペルソナの柔軟な割り当て（14種）           │
  │  ✦ セッション分離が自動                       │
  │  ✦ tmux レイアウトがコンパクト                │
  └─────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────┐
  │              弱み (Weaknesses)                │
  │                                              │
  │  ▽ 品質保証メカニズムがない                   │
  │  ▽ コードレビュープロセスがない               │
  │  ▽ 設計フェーズが簡素                         │
  │  ▽ LLM選択肢が限定的                         │
  │  ▽ フィードバックループがない                 │
  │  ▽ Living Spec の概念がない                   │
  │  ▽ 差し戻しループがない（一方向のみ）         │
  │  ▽ タスク実行後の検証が弱い                   │
  └─────────────────────────────────────────────┘
```

## 9. 統合可能性

```
理想的な統合アーキテクチャ:

  ┌──────────────────────────────────────────────────────────┐
  │           dark-part-time-job (中央オーケストレータ)        │
  │                                                          │
  │  yb CLI → 計画 → バリデーション → 注入                   │
  └──────────────────────┬───────────────────────────────────┘
                         │ yb init で注入
                         ▼
  ┌──────────────────────────────────────────────────────────┐
  │           tmux-agents-starter (実行エンジン)              │
  │                                                          │
  │  ┌─ 品質重視タスク ─→ System A（パイプライン）            │
  │  │   要件→設計→品質→実装→レビュー                        │
  │  │                                                       │
  │  └─ 速度重視タスク ─→ System B（ヤクザ階層）             │
  │      親分→若頭→若衆 並列実行                             │
  └──────────────────────────────────────────────────────────┘
```

**統合のポイント:**

| 層 | 担当 | 役割 |
|----|------|------|
| 計画・注入 | B: dark-part-time-job | `yb plan` で要件定義 → `yb init` で注入 |
| 品質重視の実装 | A: System A | フルパイプラインで品質保証付き開発 |
| 速度重視のタスク | A: System B / B 共通 | 親分→若頭→若衆で並列バッチ実行 |
| フィードバック | A: tmux-agents-starter | FB-*.md で学習を蓄積 |

## 10. 結論

### 一言で言うと

| システム | 性格 |
|---------|------|
| **A: tmux-agents-starter** | 「工場」— 品質管理付きの生産ライン |
| **B: dark-part-time-job** | 「傭兵団」— どこにでも送り込める実行部隊 |

### 最終判定

```
┌────────────────────────────┬──────────────────────────────────────┐
│         判定基準            │              勝者                    │
├────────────────────────────┼──────────────────────────────────────┤
│ 品質重視の開発              │  A: tmux-agents-starter              │
│ 手軽さ・汎用性              │  B: dark-part-time-job               │
│ 大規模プロジェクト           │  A: tmux-agents-starter              │
│ 複数リポジトリ管理           │  B: dark-part-time-job               │
│ LLMの柔軟性                 │  A: tmux-agents-starter              │
│ CLI の使いやすさ             │  B: dark-part-time-job               │
│ 学習曲線の低さ              │  B: dark-part-time-job               │
│ 継続改善能力                │  A: tmux-agents-starter              │
